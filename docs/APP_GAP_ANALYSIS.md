# Анализ текущего состояния приложения (gap analysis)

Дата анализа: 2026-04-07

## 1) Ключевой вывод

Проект функционально богатый, но сейчас это **гибрид из нескольких архитектур**, где часть модулей дублируется и не подключена в реальный runtime. Главный риск — не отсутствие отдельных экранов, а рассинхрон между:

- монолитным PWA (`index.html` + `js/*`),
- React/Vite фронтендом (`frontend/src/*`),
- Express API (`backend/src/*`),
- и «лишними» серверными файлами внутри `frontend/src/routes/*`.

Из-за этого часть возможностей «как будто есть», но фактически не исполняется в проде или ломается при сборке/поддержке.

---

## 2) Что уже реализовано и работает концептуально

- Есть рабочие доменные блоки: задачи, проекты, монтажи, заявки, чат, склад.
- Есть backend-слой на Express + Supabase.
- Есть React-приложение с базовыми страницами и роутингом.
- Есть Android-обёртка и обновление версии.

Проблема — не в отсутствии фич как таковых, а в целостности решения.

---

## 3) Что не реализовано / реализовано неполноценно

## 3.1. Нет единого «источника истины» по архитектуре

### Симптом
В репозитории одновременно присутствуют:

- монолитный web-клиент в корне,
- React-клиент в `frontend/src`,
- серверный код внутри frontend-папки (`frontend/src/index.js`, `frontend/src/routes/*`),
- отдельный backend в `backend/src`.

### Риск
- Непредсказуемый путь развития (непонятно, что деплоится и что поддерживается).
- Рост дефектов из-за правок «не в том» слое.

### Что доделать
1. Утвердить целевую архитектуру (рекомендуется: `frontend/src` + `backend/src`).
2. Зафиксировать это в README (сценарии запуска/деплоя).
3. Депрекейтнуть и затем удалить устаревшие дубли.

---

## 3.2. Backend подключён частично: не все роуты зарегистрированы

### Симптом
В `backend/src/routes` есть `purchaseRequests.js`, `materials.js`, `warehouse.js`, но в `backend/src/index.js` подключены только auth/projects/tasks/installations/users/chats.

### Риск
- API-эндпоинты физически существуют, но недоступны.
- Фронт может рассчитывать на API, которого нет в runtime.

### Что доделать
1. Зарегистрировать отсутствующие роуты в `backend/src/index.js`.
2. Добавить smoke-тесты на все обязательные endpoint-группы.
3. Уточнить базовый контракт API (OpenAPI/таблица эндпоинтов).

---

## 3.3. Ролевые ограничения местами задекларированы, но не обеспечены

### Симптом
В `materials.js` комментариями отмечено «manager only», но middleware авторизации/проверки роли на части методов отсутствует.

### Риск
- Нарушение модели безопасности (критично для складских операций).

### Что доделать
1. Проставить `authenticateToken` на весь роутер материалов.
2. Проставить `requireManager` на create/update/delete операции.
3. Добавить негативные тесты на 401/403.

---

## 3.4. Реактовый модуль чата подготовлен, но не включён в маршрутизацию

### Симптом
Есть страница `frontend/src/pages/Messenger.jsx`, но маршрута `/messenger` в `frontend/src/App.jsx` нет.

### Риск
- Функционал формально разработан, но недоступен пользователю.

### Что доделать
1. Добавить защищённый route для Messenger.
2. Добавить entry-point из Dashboard/меню.
3. Прогнать e2e путь «открытие чата → отправка сообщения».

---

## 3.5. Монорепо-скрипты в корне не соответствуют фактической структуре

### Симптом
Корневой `package.json` ссылается на `apps/*`, но в дереве используются `frontend/`, `backend/`, `js/`, `android/`.

### Риск
- «Ложный» developer experience: стандартные команды не отражают реальный запуск проекта.

### Что доделать
1. Синхронизировать workspaces/scripts с реальной структурой.
2. Добавить команды: `frontend:dev`, `frontend:build`, `backend:dev`, `backend:start`.
3. Описать это в README как единственный правильный путь запуска.

---

## 3.6. Секреты и ключи зашиты в клиентском коде

### Симптом
В корневом `index.html` присутствуют значения Firebase config, VAPID и Mapbox token в явном виде.

### Риск
- Утечка интеграционных ключей, неуправляемое использование внешних сервисов.

### Что доделать
1. Вынести конфиги в env/секреты сборки.
2. Для Mapbox и Firebase ограничить ключи по доменам/пакету.
3. Проверить репозиторий secret scanning и выполнить ротацию ключей.

---

## 4) Приоритетный план доработок

## Этап 1 (критично, 1–2 недели)

1. Архитектурное решение: что является production-клиентом.
2. Подключение всех обязательных backend-роутов.
3. Закрытие security-gap по ролям в materials/warehouse.
4. Ротация и вынос ключей из клиентского кода.

## Этап 2 (стабилизация, 2–4 недели)

1. Удаление/архивация дублей (особенно server-код внутри frontend).
2. Введение API-контрактов и smoke-тестов.
3. Нормализация скриптов и CI для каждого слоя.

## Этап 3 (развитие, 4+ недель)

1. Включение скрытых фич (например, Messenger route).
2. Единые UI/UX-стандарты между экранами.
3. Метрики: error-rate, latency API, delivery push, crash-free rate Android.

---

## 5) Мое видение целевой модели

Лучший путь для этой кодовой базы:

- **Frontend:** только React/Vite (`frontend/src`) как единый web-клиент.
- **Backend:** только `backend/src` как API-слой.
- **Legacy монолит (`index.html`, `js/*`):** либо миграция, либо явный freeze + архив.
- **Контроль качества:** обязательные проверки сборки и smoke API в CI.

Это даст предсказуемые релизы, ускорит разработку и резко уменьшит «скрытые» баги из-за дублирующихся реализаций.
